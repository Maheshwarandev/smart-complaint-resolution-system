## Core Backend Flow
1. The client sends a request to the backend.
2. server.js loads environment variables, connects to MongoDB, and starts the app.
3. app.js configures middleware, CORS, static uploads, and route mounting.
4. Routes dispatch requests to controllers.
5. Controllers interact with Mongoose models.
6. Responses are returned as JSON, and errors pass through the global handler.

---

## Phase 1: Startup & App Structure
1. package.json
   - Contains scripts for starting the server.
   - Includes Express, Mongoose, dotenv, CORS, JWT, bcryptjs, and multer.

2. server.js
   - Loads environment variables.
   - Connects to MongoDB through config/db.js.
   - Starts the app on the configured port.

3. config/db.js
   - Opens a MongoDB connection using MONGO_URI.
   - Logs success or exits on failure.

4. app.js
   - Adds JSON body parsing and CORS.
   - Serves uploaded files from /uploads.
   - Mounts /api/auth, /api/complaints, and /api/admin.
   - Adds root health-check and 404/error middleware.
   - Applies auth rate limiting.

## Phase 2: Authentication
1. models/User.js
   - Defines the schema for users.
   - Stores name, email, password, role, agentSecurityCode, createdAt.
   - Hashes passwords only when changed.
   - Supports comparePassword() and generateSecurityCode().

2. routes/authRoutes.js
   - POST /api/auth/register
   - POST /api/auth/login
   - GET /api/auth/me

3. controllers/authController.js
   - register creates a user and returns a JWT.
   - login validates credentials and returns a JWT.
   - getMe returns the currently authenticated user.

4. middleware/authMiddleware.js
   - protect verifies the JWT from the Authorization header.
   - authorize checks the authenticated role.

WHAT IS RBAC?

RBAC = Role-Based Access Control.
Each user has a role: 'user', 'agent', or 'admin'.
The role determines what routes they are allowed to call.

Role table:
  user  -> can file and manage their own complaints
  agent -> can view all complaints, update status and resolution notes
  admin -> can manage users, assign complaints, view the dashboard

HOW RBAC IS ENFORCED IN PHASE 4:

  Two middleware layers sit in front of EVERY admin route:

    router.use(protect)
      -> Checks that a valid JWT token is in the Authorization header.
      -> Decodes the token and loads the user from the database.
      -> Sets req.user so controllers know who is calling.
      -> If token is missing or invalid -> 401 Unauthorized.

    router.use(authorize('admin'))
      -> Reads req.user.role.
      -> If role is not 'admin' -> 403 Forbidden.
      -> If role is 'admin' -> continues.

  Both are applied once at the TOP of adminRoutes.js using router.use().
  Every route below is automatically covered.

---

1. controllers/adminController.js
   - Exports 6 functions

   getDashboard (GET /api/admin/dashboard)
     - Runs 6 MongoDB countDocuments() queries simultaneously with Promise.all()
     - WHY Promise.all()? Running queries one-by-one is slower.
       Promise.all() starts all 6 at the same time and waits for all results.
     - Counts: totalUsers, totalAgents, totalComplaints, open, inProgress, resolved

   getAllUsers (GET /api/admin/users)
     - User.find({ role: 'user' }).sort({ createdAt: -1 })
     - Returns only regular users, sorted newest first

   getAllAgents (GET /api/admin/agents)
     - User.find({ role: 'agent' }).sort({ createdAt: -1 })
     - Returns only agents — useful when the admin needs to pick one to assign a complaint to

   updateUserRole (PUT /api/admin/users/:id/role)
     - Validates role is one of: user, agent, admin
     - Blocks admin from changing their own role (safety guard)
     - Finds user, sets user.role, calls user.save()
     - WHY .save()? Runs Mongoose enum validators.
       findByIdAndUpdate() skips validators — risky for enum fields.

   assignComplaint (PUT /api/admin/complaints/:id/assign)
     - Body: { "agentId": "..." }
     - Validates agentId is provided
     - Confirms the agentId belongs to a real user with role === 'agent'
     - Sets complaint.assignedTo = agentId
     - Auto-sets complaint.status = 'In Progress'
     - WHY auto-set status? Once assigned, the complaint is actively being handled.
       The admin should not need a second API call just to update the status.
     - Populates assignedTo after save so response shows name + email

   deleteUser (DELETE /api/admin/users/:id)
     - Blocks admin from deleting themselves (safety guard)
     - Finds the user by ID
     - Deletes all their complaints first: Complaint.deleteMany({ user: id })
     - WHY delete complaints first? Without this, complaints remain in the DB
       with a user field pointing to a deleted user — a dangling reference.
     - Deletes the user: user.deleteOne()
     - WHY deleteMany() and not a loop? One query is faster than N queries.

2. routes/adminRoutes.js
   - router.use(protect)             -> covers all routes below
   - router.use(authorize('admin'))  -> covers all routes below

   GET    /api/admin/dashboard              -> getDashboard
   GET    /api/admin/users                  -> getAllUsers
   GET    /api/admin/agents                 -> getAllAgents
   PUT    /api/admin/users/:id/role         -> updateUserRole
   PUT    /api/admin/complaints/:id/assign  -> assignComplaint

ADMIN FLOW DIAGRAM

  CLIENT (Postman / React)
    |  Authorization: Bearer <admin-token>
    |  e.g. GET /api/admin/dashboard
    v
  app.js -> express.json() -> app.use('/api/admin', adminRoutes)
    |
    +- router.use(protect)
    |     1. Extract token from Authorization header
    |     2. jwt.verify(token, JWT_SECRET) -> decoded = { id: ... }
    |     3. User.findById(decoded.id) -> req.user
    |     4. If any step fails -> 401 Unauthorized (stopped here)
    |
    +- router.use(authorize('admin'))
    |     5. Check req.user.role === 'admin'
    |     6. If role !== 'admin' -> 403 Forbidden (stopped here)
    |
    +- Controller function runs
          7. Executes the admin logic (dashboard, assign, delete, etc.)
          8. Returns JSON response

---

ADMIN FLOW: Promote User to Agent

  1. Admin logs in -> POST /api/auth/login -> receives token
  2. Admin calls GET /api/admin/users -> sees list of users with their IDs
  3. Admin calls PUT /api/admin/users/<userId>/role
     Body: { "role": "agent" }
  4. updateUserRole validates -> updates -> saves -> returns updated user

ADMIN FLOW: Assign Complaint to Agent

  1. Admin calls GET /api/admin/agents -> sees list of agents with their IDs
  2. Admin finds an open complaint (via GET /api/complaints or dashboard)
  3. Admin calls PUT /api/admin/complaints/<complaintId>/assign
     Body: { "agentId": "<agentId>" }
  4. assignComplaint confirms agent, updates complaint, auto-sets 'In Progress'

ADMIN FLOW: Delete User

  1. Admin calls GET /api/admin/users -> gets list with IDs
  2. Admin calls DELETE /api/admin/users/<userId>
  3. deleteUser removes their complaints first, then removes the user
